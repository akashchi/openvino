#### Explanation

The PR adds new snippets repacking passes for x64/AArch64. In the x64 path, `RepackMatMulWeights::get_dst_cpu_desc()` builds the blocked destination descriptor from `shape.getStaticDims()`, but the upstream source shape can still be dynamic when the model carries dynamic dimensions through a reorder/partial-shape path. That makes the `Shape::getStaticDims()` assertion fire during compilation. The failure is broad across CPU layer tests and model-hub tests because they all exercise the same compile-time repacking flow.

#### Confidence

High — the same assertion appears across multiple CPU jobs, and the only changed code in scope unconditionally requests static dims inside the new repacking logic.

### Reproduction Steps

Run the affected CPU model-hub or layer tests with a dynamic-shape attention model on the PR branch; the compile step aborts with the same `cpu_shape.h:128` assertion.

### Recommended Actions

- [ ] Guard repacking so dynamic shapes skip this path or fall back to the existing non-repacked flow.
- [ ] If repacking must remain enabled, derive the destination descriptor from a guaranteed-static weight shape, not from the dynamic partial shape.
- [ ] Add a regression test covering a dynamic-shape CPU model that previously reached `cpu_shape.h:128`.

### Prevention Strategies

Avoid calling `getStaticDims()` in compile-time passes unless the code first proves the shape is static. Add a regression test whenever a new CPU transformation touches dynamic-shape compile paths.

### AI Team Self-Improvement

When modifying CPU transformation passes, explicitly check whether any shape flowing into a `getStaticDims()` call can be dynamic. If yes, gate the pass or preserve the original static weight shape, and add a dynamic-shape regression test.
